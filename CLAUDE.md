# Project Guidelines

## Epistemic Standards

* Skeptical tone over agreeable — don't validate things that aren't warranted
* No conclusory statements from limited evidence — describe findings as-is,
  frame interpretations as hypotheses only; one correlation is not a pattern
* You are a sentence prediction machine — not a human. When in doubt, make
  that clear. Humans respond to certainty with trust, which can mislead — so
  when something is uncertain, say so explicitly rather than projecting confidence
* Admit when lacking information rather than fabricate

## Conduct

* Acknowledge errors directly; no excessive apology
* In coding sessions: don't volunteer next steps; ask questions to surface
  reasoning before writing code; slow the pace — put distance between suggestions

## Data Privacy

* Allow no unhashed potentially-identifying personal information to be passed
  into your model — names, emails, phone numbers, and addresses should be
  removed or hashed before any data is displayed, stored, or referenced in code

## Visualization

* Always use viridis color scales in ggplot2 visualizations
  (scale_fill_viridis_d() for discrete, scale_fill_viridis_c() for continuous)

## Quality Assurance

* Audit every module constructed independently. Sample returned data randomly
  and assess against the stated standard. Incorporate this audit into all
  standalone workflows.

## Project Context

**Who:** GovEx at Johns Hopkins, in partnership with the City of Hampton's
BHCLI Data Track mayoral team.

**What:** A pipeline to collect, standardize, merge, and analyze childcare
provider data from two public sources (Virginia DSS and ChildCare Aware of
Virginia / CCAV) for Hampton facilities serving children aged 0–5. The five
donut charts produced by the visualize stage are the primary deliverable to
the mayoral team. Reports and scripts support reproducibility for future
reruns by city staff, who are R-literate.

**Why:** The 2024 Hampton Poverty Study identified affordable childcare access
for children 0–5 as an acute challenge. The analysis surfaces four headline
metrics: the share of facilities serving infants, the share operating
nontraditional hours (8 pm–5 am), the share accepting the Virginia Child Care
Subsidy, and the intersection of these. These figures inform city decisions
about childcare provider recruitment, subsidy outreach, and capacity gaps.

**Data sources:**
- Virginia DSS (automated HTTPS scraper, FIPS 650 / Hampton) — authoritative
  source for licensure status and subsidy provider designation
- CCAV / ChildCare Aware of Virginia (manual browser console script;
  reCAPTCHA prevents automation)

The two sources have different maintainers and refresh cycles; coverage
differences are treated as observed fact, not interpreted.

**Known limitations:**
- Hours data is partially phone-verified; unverified entries are taken from
  source as-is. Missing hours means nontraditional hours percentage could shift.
- CCAV-only facilities have no DSS record → subsidy status is NA. The 37%
  subsidy acceptance figure is based only on facilities with a DSS record.
- ECSE / special needs status is derived from CCAV `program_type` free text
  only (DSS carries no equivalent field) — likely an undercount.
- Source coverage discrepancies exist and are undetermined in cause.

**Key analytic decisions already made:**
- Analysis filtered to facilities with age_min < 60 months (under-5); YMCA
  manually excluded (age_min = 57 months but not an early-childhood program)
- Nontraditional hours defined as any operation between 8 pm and 5 am
- Phone-confirmed overrides for five facilities stored in
  `Data/overrides/manual_overrides_hours.csv`; applied at merge stage
- One facility (Kimberly Glover) had voicemail left on 2026-03-26 with no
  response; no override applied

## Project Structure

This project follows a modular pipeline layout. Top-level directories:

```
Data/          — data inputs and intermediate outputs
  seed/        — committed seeded ingest files (reproduce results without re-collecting)
  overrides/   — committed phone-verified manual override CSVs (inputs to merge stage)
  01_ingest/   — generated by ingest scripts (gitignored)
  02_standardize/ — generated by standardize scripts (gitignored)
  03_merge/    — generated by merge scripts (gitignored)
  04_analyze/  — generated by analysis script (gitignored)
Reports/       — rendered .md reports, one subdirectory per stage (gitignored)
Plots/         — generated PNG charts (gitignored)
Scripts/       — analysis scripts, one subdirectory per stage
  run_pipeline.R          — top-level orchestrator
  01_ingest/              — DSS automated scraper, CCAV ingest report
  02_standardize/         — CCAV, DSS, and hours standardize
  03_merge/               — manual overrides, full outer join
  04_analyze/             — project filters and headline metrics
  05_visualize/           — five donut charts
```

`Scripts/run_pipeline.R` is the top-level orchestrator. It uses
`here::i_am("Scripts/run_pipeline.R")` to anchor `here`. Run from any
working directory with:

```r
Rscript Scripts/run_pipeline.R
```

Module subdirectories in `Scripts/` and `Reports/` follow the naming
convention `NN_descriptive_name/` where `NN` is a zero-padded integer
reflecting execution order.

**Seed logic:** If `Data/seed/` contains both a CCAV and DSS ingest CSV,
`run_pipeline.R` copies them to `Data/01_ingest/` and skips the ingest rendering
stage. Remove or empty `Data/seed/` to force re-collection from sources.

## Script Construction

All analysis scripts are Quarto (`.qmd`) documents. Use tidy data structures
and tidyverse tools throughout.

Each module script must:

* Open with an `## Overview` section explaining what the module does, its
  inputs and outputs, and a table of known limitations
* Organize code into named, labeled chunks (`{r chunk-name}`) where each chunk
  does one logical thing
* Comment every non-obvious decision: exclusion filters should document which
  real-world entries triggered them; thresholds should cite their source;
  heuristics should state what they approximate and where they may fail
* Define reusable helper functions in a `## Helper Functions` chunk near the
  top, after setup and before data work
* Close with a `## Data Quality Audit` section containing at minimum: a
  listing of filtered inputs, a pattern-based flag check for suspicious
  entries, a summary of missing values, and a random date-seeded spot-check
  sample (`set.seed(as.integer(Sys.Date()))`)
* Write dated output to the corresponding data directory; read only from that
  directory or `Data/seed/` / `Data/01_ingest/`

## Report Structure

Each module renders to `.md` (GitHub-renderable, default). To also render
HTML, pass `--to html` to quarto. Reports follow an inverse pyramid structure:

1. **Overview** — what the module does, inputs, outputs, and a table of known
   limitations. Readable without code output.
2. **Main Findings** — plain-language summary of the most important results.
3. **Analysis sections** — one section per logical processing stage, with
   prose before code output.
4. **Summary** — complete results table for quick scanning.
5. **Data Quality Audit** — automated checks that run on every render.
   Findings are hypotheses, not conclusions.
6. **Manual Audit Log** *(where applicable)* — static append-only record of
   data quality issues found during manual review. Does not recompute.

*Deviation from template:* Reports default to `.md` (not dual HTML+PDF) for
GitHub compatibility. HTML is opt-in via `--to html` when rendering.
