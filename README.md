# Hampton Childcare Provider Pipeline

This pipeline collects, standardizes, merges, and analyzes childcare provider data for the City of Hampton, Virginia, drawing from two public sources: the Virginia Department of Social Services (DSS) and ChildCare Aware of Virginia (CCAV).

Developed by [GovEx at Johns Hopkins University](https://govex.jhu.edu) in partnership with the City of Hampton. Claude (Anthropic) was used in the production of this code.

|---|---|---|

### Run the pipeline

Seeded ingest data from April 2026 is included in `Data/seed/` so you can run the full pipeline immediately without re-collecting data:

```r
Rscript Scripts/run_pipeline.R
```

Reports appear in `Reports/` as `.md` files. Plots appear in `Plots/`.

> **To update with fresh data**, re-run the ingest stage (see [Stage 1: Ingest](#stage-1-ingest)). The DSS script is fully automated; the CCAV script requires a brief manual step in Chrome. Remove or empty `Data/seed/` before running to force the pipeline to re-collect from sources.

> **Viewing reports on GitHub:** `.md` files render natively in GitHub.

## Pipeline Stages

### Stage 1: Ingest

> **Skip this stage if you're reproducing the existing results.** `Data/seed/` contains seeded files from April 2026 (`CCAV_ingest_2026-04-08.csv` and `DSS_ingest_2026-04-03.csv`). `run_pipeline.R` detects these and copies them to `Data/01_ingest/` automatically, bypassing the ingest scripts. Remove or empty `Data/seed/` only when you want to collect fresh data.

Two sources, two different collection methods.

#### DSS (automated)

The pipeline renders `Scripts/01_ingest/dss_ingest.qmd` automatically when no seed files are present.

Scrapes the [Virginia DSS facility search](https://www.dss.virginia.gov/facility/search/cc2.cgi) for all Hampton providers (FIPS 650) using `httr` + `rvest`. No credentials required. Takes ~30 seconds.

> **Note on SSL:** Virginia's state server uses a certificate chain not trusted by R's default CA bundle. SSL verification is intentionally disabled for this URL only.

#### CCAV (manual — requires Chrome)

The CCAV search tool requires solving a reCAPTCHA, so this step cannot be fully automated.

1. Open [https://stage.worklifesystems.com/parent/25](https://stage.worklifesystems.com/parent/25) in Chrome
2. Click **Guest Account**, complete the reCAPTCHA
3. Confirm the results show Hampton providers
4. Open Chrome DevTools: `Cmd+Option+J` (Mac) or `Ctrl+Shift+J` (Windows)
5. Paste the entire contents of `Scripts/01_ingest/ccav_ingest.js` into the Console and press Enter
6. The script pages through all results and downloads `CCAV_ingest_YYYY-MM-DD.csv`
7. Move the downloaded file to `Data/01_ingest/`

Then re-run `Scripts/run_pipeline.R` — it will detect the CCAV file and proceed.

### Stage 2: Standardize

Three scripts normalize each source's field formats and produce merge keys.

| Script | Input | Output |
|---|---|---|
| `ccav_standardize.qmd` | `Data/01_ingest/CCAV_ingest_*.csv` | `Data/02_standardize/CCAV_standardize_*.csv` |
| `dss_standardize.qmd` | `Data/01_ingest/DSS_ingest_*.csv` | `Data/02_standardize/DSS_standardize_*.csv` |
| `hours_standardize.qmd` | Both standardized CSVs | `Data/02_standardize/hours_long_*.csv` |

Each script also renders a `Reports/02_standardize/*.md` quality-check report.

Key transformations:
- Age strings (e.g. `"1 Year 4 Months - 12 Years 11 Months"`) → `age_min_months`, `age_max_months`
- Hours strings (e.g. `"6:00 am - 5:00 pm, Monday - Friday"`) → `days`, `open_time`, `close_time` (24h HH:MM)
- Names and addresses → normalized `name_key` and `address_key` for joining

---

### Stage 3: Merge

| Script | What it does |
|---|---|
| `manual_override.qmd` | Writes phone-confirmed hours corrections to `Data/overrides/manual_overrides_hours.csv` |
| `merge_providers.qmd` | Full outer join of CCAV and DSS on `name_key`; applies overrides; flags discrepancies |

`merge_providers.qmd` produces:
- `Data/03_merge/providers_merged_*.csv` — one row per facility
- `Data/03_merge/providers_hours_merged_*.csv` — one row per facility × day
- `Reports/03_merge/merge_providers.md`

To add a new phone-confirmed hours correction, add rows to the `hours_overrides` tribble in `Scripts/03_merge/manual_override.qmd`. Each `name_pattern` must match at least one `name_key` in the merged table or the pipeline will stop with an error.

---

### Stage 4: Analyze

`Scripts/04_analyze/providers_analysis.qmd` applies project-specific filters (under-5 facilities only, YMCA excluded), computes nontraditional hours and weekend open flags, and produces:
- `Data/04_analyze/providers_analysis_*.csv`
- `Reports/04_analyze/providers_analysis.md`

---

### Stage 5: Visualize

`Scripts/05_visualize/visualizations.qmd` generates five donut charts saved to `Plots/` and a summary document in `Reports/05_visualize/visualizations.md`.

To render as HTML instead of markdown:

```bash
quarto render Scripts/05_visualize/visualizations.qmd --to html --output-dir Reports/05_visualize
```

## Configuration

`config.R` at the project root defines `prune_dated()`, which automatically keeps the two most recent dated versions of every output file and deletes older ones. No other configuration is needed for a standard run.

Optional environment overrides (copy `.env.example` to `.env`):

```bash
RSCRIPT=/path/to/Rscript   # override Rscript location
QUARTO=/path/to/quarto     # override quarto location
CHROME=/path/to/chrome     # needed only for headless PDF rendering
```

## Data Sources

| Source | URL | Method | Frequency |
|---|---|---|---|
| Virginia DSS | [dss.virginia.gov/facility/search/cc2.cgi](https://www.dss.virginia.gov/facility/search/cc2.cgi) | Automated R scraper | On demand |
| CCAV | [stage.worklifesystems.com/parent/25](https://stage.worklifesystems.com/parent/25) | Manual browser script | On demand |

Both sources are publicly accessible. No API keys or credentials are required.

### Phone-verified overrides

Hours confirmed by phone call are stored in `Scripts/03_merge/manual_override.qmd`. To add a new verified facility, add rows to the `hours_overrides` tribble following the existing pattern. See the Manual Audit Log in that document for the verification history.
